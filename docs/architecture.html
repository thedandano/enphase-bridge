enphase-bridge — Architecture

Self-hosted Rust daemon that polls an Enphase IQ Gateway over HTTPS, aggregates solar energy into 15-minute windows, and serves them via a local REST API backed by SQLite. Three concurrent Tokio tasks race under a single tokio::select!: the collector poll loop, the Axum API server, and the TOU auto-refresh loop.

TokenManager src/auth/token_manager.rs pure startup-once
Base64-decodes the Enphase cloud JWT payload locally to read the exp claim. No network call — ES256 signature is not verified. Process exits if token is expired; warns if expiry is within 30 days.

GatewayClient src/collector/gateway_client.rs
Manages session cookie via check_jwt(). At scheduler startup calls probe_meters() to confirm the net-consumption meter is present and enabled — halts with GatewayError::MissingMeter if absent (no silent fallback). Each poll calls get_meter_readings(); auto-retries with a fresh check_jwt() on any 401. On window boundary calls get_inverter_snapshots(). Deserializes JSON → MeterReadings / Vec<InverterReport>.

Scheduler src/collector/scheduler.rs
Tokio interval loop. Startup: calls check_jwt() then probe_meters(); loads persisted anchor reading from config_store. Each tick: polls gateway via GatewayClient, calls window_boundary() to detect 15-min crossings, delegates to compute_delta() and parse_snapshots(), writes to storage, and persists the new anchor Wh values.

Window crossed — write path
WindowAggregator (src/collector/window_aggregator.rs) compute_delta(prev, curr): subtracts lifetime Wh counters; derives wh_consumed = produced + import − export; clamps all deltas to ≥ 0.

InverterSnapshot parser (src/inverter/snapshot.rs) parse_snapshots(reports, window_start): marks each micro-inverter online if last-report age < 20 min (OFFLINE_THRESHOLD_SECS = 1200), else zeros watts.

Scheduler writes EnergyWindow → energy_window table; writes Vec<MicroinverterSnapshot> → microinverter_snapshot table; persists new anchor (timestamp + cumulative Wh) to config_store.
Mid-window tick — anchor frozen
No window written. Current reading is held in memory but the persisted anchor is not updated — it stays pinned to the last boundary reading so the next window delta is computed correctly.
Gateway poll error
poll_error logged; tick skipped via continue. Loop never exits. 401 triggers automatic check_jwt() re-auth transparent to the loop.

TrueUp Calculator src/trueup/calculator.rs pure
Receives &TouRateSchedule + &[EnergyWindow]; returns CalculatorResult. Classifies each window into Peak / Off-Peak / Super Off-Peak by ranking OpenEI rate periods: highest rate → Peak; lowest rate (when ≥ 3 distinct periods) → Super Off-Peak; rest → Off-Peak. Stable sort preserves index order on rate ties. No I/O.
The API server shares the SQLite pool with the collector. WAL mode allows concurrent readers alongside the single writer, so reads never block writes.
TOU auto-refresh loop · every 7 days
TOU Refresh Loop src/tou/refresh.rs background
At startup: checks if the stored schedule is missing or older than 7 days; if so, fires an immediate refresh. Then enters an infinite loop sleeping 7 days between refreshes. Errors are logged but the loop never exits — failures are non-fatal.
fetches when stale · GET /utility_rates?eia=…
OpenEI URDB API api.openei.org · external
Third-party utility rate database. Returns residential TOU rate schedules as JSON with energyweekdayschedule, energyweekendschedule, and energyratestructure.
rate JSON · matched by rate_label name field
OpenEiClient src/tou/openei_client.rs
Fetches GET /utility_rates?eia=…&api_key=…; finds the item matching rate_label by name; picks the most recent version by startdate. Returns a FetchedSchedule with raw rate JSON. Base URL overrideable via with_base_url() for tests.
FetchedSchedule inserted into DB
tou_schedule storage src/storage/tou_schedule.rs
Inserts new versioned rows into tou_rate_schedule; never deletes existing rows. query_latest(pool, rate_label) used by handlers and probe.
tou_rate_schedule
Two distinct freshness thresholds: the refresh loop fires at 7 days; the health endpoint and startup probe warn at 90 days. This prevents noisy warnings during the normal 7-day cadence.
TOU startup probe · once at boot
probe_tou_schedule src/tou/probe.rs once
Called from main.rs before the tokio::select! block. Queries tou_rate_schedule for the latest row matching rate_label. Emits structured log event tou_schedule_ok (info) if fresh; tou_schedule_stale (warn) if absent or older than 90 days. Warn-only — never exits the process.
The probe is diagnostic only — it does not trigger a fetch. Re-fetching is handled exclusively by the TOU refresh loop (7-day threshold). The two-threshold design (7 days vs 90 days) lets ops teams notice genuine stale schedules without false alarms during the normal refresh window.
Process orchestrator · tokio::select!
main.rs — tokio::select! src/main.rs
Sequential startup: load config → connect SQLite + run migrations → decode JWT → create AppState + GatewayClient + Scheduler → call probe_tou_schedule. Then races three async futures:
  1. API server — api::server::serve()
  2. Collector loop — scheduler.run()
  3. TOU refresh loop — tou::refresh::run_tou_refresh_loop()
If any task returns, select! drops the others. In normal operation all three run indefinitely.
All three tasks share one SqlitePool clone. WAL mode ensures the single write connection (collector) does not block the read connections used by the API and TOU loops.

Isolated (✓ full) — the component can be replaced by editing one file with no cascading changes to other modules.
Pure fn (✓ pure fn) — stateless function with no I/O; trivially replaceable and unit-testable with a plain assert_eq!.
Orchestration hub — intentionally calls multiple other components; this is expected and correct for coordinators. Replacing them requires coordinating all downstream contracts.
