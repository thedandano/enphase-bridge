# Implementation Plan: Enphase Gateway Data Service

**Branch**: `001-enphase-gateway-api` | **Date**: 2026-04-26 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/001-enphase-gateway-api/spec.md`

---

## Summary

Build a Rust daemon that polls the Enphase IQ Gateway local API every 60 seconds, aggregates data into 15-minute energy windows (system-level and per-microinverter), stores it in a local SQLite database, and exposes a read-only HTTP API. Adds SDGE TOU rate ingestion from OpenEI URDB and a NEM true-up cost estimator. Delivered as five independent vertical slices, each a functional MVP increment.

---

## Technical Context

**Language/Version**: Rust stable (1.87+)
**Primary Dependencies**:
- `tokio 1.40+` — async runtime
- `axum 0.8+` — HTTP API server
- `reqwest 0.12+` — HTTP client (gateway + OpenEI; separate instances)
- `sqlx 0.8+` (sqlite feature) — storage with compile-time query verification
- `jsonwebtoken 9.3+` — JWT parsing and expiry checking
- `figment 0.10+` — TOML + env var config
- `tracing 0.1.40+` + `tracing-subscriber` — structured JSON/logfmt logging
- `serde 1.0` + `serde_json 1.0` — serialization

**Storage**: SQLite (embedded, WAL mode, single writer) — `energy.db` on host filesystem

**Testing**: `cargo test`, `tokio::test` for async integration tests

**Target Platform**: Linux/macOS — Raspberry Pi 4+, home server, Mac Mini

**Project Type**: Long-running daemon + embedded HTTP API server

**Performance Goals**:
- API time-range query (30 days): < 500ms (SC-003)
- True-up computation (12 months): < 2 seconds (SC-007)
- Per-inverter snapshot availability: within 30s of window boundary (SC-008)
- Collection delay beyond poll interval: ≤ 5 seconds (SC-001)

**Constraints**: Low memory footprint (Raspberry Pi compatible); no external database process; no cloud dependency for core operation; gateway TLS cert is self-signed (disable cert verify for gateway client only)

**Scale/Scope**: Single user, single gateway, ~35K energy windows/year + ~35K × N_inverters microinverter rows/year (N typically 10–30)

## Vertical Slices (Delivery Order)

Each slice is independently demonstrable and deployable. No slice begins until the prior slice passes its independent test.

### Slice 1 — Gateway Auth + Basic Poll (US2 complete)

**Delivers**: The service starts, authenticates with the gateway, and logs a successful response from `/ivp/meters/readings`. No storage yet.

**Demo**: Start the binary → see structured log line confirming gateway response with `wNow` value.

**Modules**: `config`, `auth/token_manager`, `collector/gateway_client`

---

### Slice 2 — 15-Minute Window Aggregation + SQLite Storage (US1 complete)

**Delivers**: Headless data logger. Polls every 60s, computes 15-min window deltas at boundary, writes `energy_window` rows to SQLite. No API yet.

**Demo**: Run for 15+ minutes → query `energy.db` directly and see rows with `wh_produced`, `wh_consumed`, `wh_grid_import`, `wh_grid_export`.

**Modules**: `collector/window_aggregator`, `collector/scheduler`, `storage/db`, `storage/energy_window`

---

### Slice 3 — Per-Microinverter Snapshots (FR-010 complete)

**Delivers**: At each 15-min boundary, queries `/ivp/peb/devstatus` and writes `microinverter_snapshot` rows. Inverter serial numbers loaded from `/inventory.json` at startup.

**Demo**: Run for 15+ minutes → query `microinverter_snapshot` and see one row per inverter per window boundary.

**Modules**: `inverter/snapshot`, `storage/inverter_snapshot`

*Can be developed in parallel with Slice 2 after Slice 1 is complete.*

---

### Slice 4 — HTTP Read API (US3 complete)

**Delivers**: `axum` HTTP server serving energy window and inverter snapshot queries. Also `GET /api/health`.

**Demo**: `curl http://localhost:8080/api/energy/windows/latest` returns JSON. `curl http://localhost:8080/api/inverters/snapshots/window/{ts}` returns per-inverter data.

**Modules**: `api/server`, `api/handlers/energy`, `api/handlers/inverters`, `api/handlers/health`

*Depends on Slices 2 and 3 (data must exist to query).*

---

### Slice 5 — OpenEI TOU Ingestion + True-Up Calculator (US4 complete)

**Delivers**: `POST /api/tou/refresh` fetches SDG&E TOU schedule from OpenEI URDB and stores it. `GET /api/trueup/estimate` computes NEM true-up using stored energy data and TOU rates.

**Demo**: `curl -X POST http://localhost:8080/api/tou/refresh` → schedule stored. `curl "http://localhost:8080/api/trueup/estimate?start=...&end=..."` returns breakdown by TOU period with net cost/credit.

**Modules**: `tou/openei_client`, `storage/tou_schedule`, `trueup/calculator`, `api/handlers/trueup`, `api/handlers/tou`

*Depends on Slice 4 (API server must be running).*

---

### Slice 6 — Web Dashboard (US5, P3 stretch — not in v1 scope)

Out of scope for this plan. The API from Slice 4 is designed to be dashboard-consumable. A future plan will address the dashboard separately.

## Cross-Cutting Concerns (all slices)

- **Logging**: Every module uses `tracing::info!/warn!/error!` macros. JSON formatter via `tracing-subscriber`. Token values MUST NOT appear in log output.
- **Error handling**: All `Result` errors bubble up via `?` and are caught at task boundaries with structured log entries. No `.unwrap()` in production paths.
- **Tests**: Each slice includes unit tests for pure logic and integration tests for I/O paths. Tests are written first (Red) before implementation (Green).
- **CI**: `cargo clippy -- -D warnings`, `cargo fmt --check`, `cargo test` must all pass.
- **Security**: `gateway_client` disables TLS cert verification only. `openei_client` uses normal TLS. Gateway token never logged.
